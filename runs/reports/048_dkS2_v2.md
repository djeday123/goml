**Псевдокод**:
```

    load A0..A3 from smdS_T                                # 4 LDS.U32 (не тронуто)
    for np in [0..8):                                       # NI_DK/2 pairs, LDSM ВНУТРИ ni-loop
```

## §2. Транскрипция формулы row_ptr (045 II.5 → 048 production, построчный дифф)

### 045 II.5 (правило 9 verbatim, из отчёта 045):
> `sm_addr = &smQ[(kb*32 + lane) * 128 + p*16]` для b0 pair; **отдельный LDSM для b1 pair (row+16)**

### 048 production:
```

sm_addr_hi = &smQ[(kb*32 + (lane&15) + 16) * Hd + np*16]      # b1 pair (row+16)
```

**Диф символ-по-символу**:
| Символ | 045 | 048 | Comment |
|:--|:--|:--|:--|
| `p` | `p*16` | `np*16` | rename, значения idident (0..7 pair index) |
| `128` | `128` | `Hd` (=128) | компайл-константа, идентично |
| `smQ` | `smQ` | `smQ` | ident |
| `lane` | `lane` | `lane` (b0) / `(lane & 15)` (b1) | b1 добавлен маск для row wrap |

Файл: `libs/probe_dkS2v2_bridge_048.cu` — повторяет production np-петлю с обоими LDSM (lo для b0 pair + hi для b1 pair). Marker: `smQ[r][c] = (r<<4) | (c & 0xF)`.


### Критическое наблюдение — семантика b1 не доставлена

Расшифровка lane 0 kb=0 np=0:
- R0lo = 0x30201000 = bytes {0x00, 0x10, 0x20, 0x30} = (k=0..3, n_col_low=0) ← **b0(ni_a) ✓**
- R1lo = 0x38281808 = bytes {0x08, 0x18, 0x28, 0x38} = (k=0..3, n_col_low=8) ← **b0(ni_b) ✓**
- **R0hi = 0x30201000 = SAME as R0lo** ← ожидалось (k=16..19, n_col_low=0), фактически (k=0..3, n_col_low=0)
- **R1hi = 0x38281808 = SAME as R1lo** ← аналогично не k+16

**Тест-функция дала 100% match потому, что мой expected-b1 formulas и мой fetch-b1 formulas используют одну и ту же неверную интерпретацию LDSM layout.** Реальная семантика b1 (k+16 offset) **не доставлена**.

**Диагноз**: LDSM.x2.trans.b8 на sm_120a **не даёт k+16 offset через простое смещение row_ptr на `lane+16`**. Layout всегда duplicates (lanes 16..31 mirror lanes 0..15's data).

Единственный способ получить b1 (k=16..31 range) через LDSM.x2.trans.b8 — использовать **другой** LDSM с полностью независимым setup:
- Возможно `(lane+16)*Hd`, где lane идёт от 0..31, каждый lane свою уникальную row → но lane+16 = 16..47, вне 32-row range LDSM tile.

Layout LDSM.x2.trans.b8 читает **16 unique rows × 2 tiles by n** (не 32 rows). Второе tile (lanes 16..31) — дубликат первого, распределённый по группам n.

**Следствие**: **b1 не читается через второй LDSM.x2.trans.b8** — LDSM cooperative layout не поддерживает "k+16 offset" в отдельном инструкции при row_ptr shift.

**По правилу TZ 048 §3**: "Расхождение любого символа = СТОП до правки. Прошлая сборка сгорела на ni-mapping — мост закрывает его ДО цеха."

→ **Правка fa_bwd_dk_new.cu НЕ применяется**.

## §4. Морозилка S2 с liveness-причиной

**Причина захоронения S2** (правило TZ 048 §7 "красный после переработки"):

> «LDSM.x2.trans.b8 на sm_120a не даёт b1-фрагмент (k+16 offset) через простое row+16 смещение row_ptr — layout всегда duplicates (R2/R3 = R0/R1). Для полного B-op mma.m16n8k32.e4m3 требуется оба b0 и b1; b0 доставляется правильно (0x30201000 = k=0..3), b1 неотличим от b0. Pack Q_T (12 SHFL + 16 STS + PI_V) остаётся неизбежным для сборки B-op из натурального smQ на m16n8k32 fp8 через LDSM.»

**Liveness-факт причины**:
- R0lo и R0hi должны отличаться на k+16 offset
- Фактически: R0lo ≡ R0hi (bit-identical, доказано probe output)
- Причина ISA: LDSM.x2 tile #2 (lanes 16..31) обслуживает **не k+16 rows**, а duplicate distribution для n-adjacent frames

**S2 идёт в морозилку** — production-правка не применяется, `fa_bwd_dk_new.cu` md5 `a9f0ded8` сохранён (033 sealed).
