## Могила (a) smQ double-buffer

**Причина смерти в архиве**: "headroom 2560 B недостаточно для 8192 B smQ; MIO-bound = вход/выход через LDS/STS, добавление буфера просто увеличит SMEM без ALU-выгоды".

**Проверка причины после 040 + E-правки**:
- **Headroom SMEM после E-cut**: **8704 B** (Vugar-акт 038).
- **smQ**: 8192 B (Br × Hd = 64 × 128 bytes).
- **8192 B ≤ 8704 B** ✓ headroom **сейчас достаточен по объёму**.
- **MIO больше не bottleneck** (8.86%), причина смерти изменилась.

**НО**: smQ живёт в **3-way alias-union** `smQ_region` (union: smQ / smdS_stage / smP_T), sequential phases:
- Phase A: smQ (Step B MMA-A reads)
- Phase C: smdS_stage (Step E scatter, Step F drain reads)
- Phase D: smP_T (Step G writes, Step H MMA_dV reads)

**Бумага "куда префетчить Q[qt+1]"**:
- Q[qt+1] cp.async prefetch должен идти во **вторую копию smQ**, живущую в headroom 8704 B.
- Требует **отдельного SMEM буфера** (не alias). Занимает 8192 B ≤ 8704 ✓.
- **Хореография alias-union НЕ трогается** (класс #7 и другие остаются на текущей раскладке).
- **Барьеры сохраняются**: любой сдвиг t3/t13 требует **отдельного ТЗ с racecheck** (правило 13).

**Диагноз могилы (a)**: **причина смерти "MIO-bound" ИСЧЕЗЛА** после 040. Причина "headroom 2560 B" ИСЧЕЗЛА после E-правки 038. **КАНДИДАТ**.

- **Механизм**: cp.async prefetch Q[qt+1] в headroom → скрытие LDG latency для Step A cp.async wait.
- **Потолок по механизму**: если long_scoreboard 6.73% is Q/dO cp.async wait — префетч потенциально сокращает эту долю. Верхняя граница ~6.73% × 25 мс = ~1.7 мс absolute потенциал. Реальный потенциал МЕНЬШЕ (нельзя убрать всю LDG latency).
- **Риск**: (i) новый барьер для prefetch-consume ordering — потребует racecheck; (ii) увеличение occupancy давления — currently 2 blk × slot 42496 = 84992 из 102400; после +8192 = 51392 slot per blk × 2 = 102784 > 102400 → **потеря блока**. Проверка: 41472 + 8192 = **49664 B/blk**, slot (49664+1024) = 50688, alignment 256 = 50688 (уже кратно). 2 × 50688 = 101376 ≤ 102400 ✓ **пройдено** (тонкий запас 1024 B).
- **Оценка ptxas**: +5..+10r адр-арифметики для двойного буфера. Люфт 252→256 держит 2 blk (66 запас).

## Могила (b) smL/smD double-buffer +512 B

**Причина смерти в архиве** (009 эпоха, кандидат (c)): "smL+smD = 512 B, prefetch следующего qt в +512 B — обмен курс не окупает MIO-headroom (общий MIO 42% dominates, +512 B не съест cost)".

**Проверка**:
- L/D reads: класс #3, 4 LDS.32 fp32 per lane per qt — минимальная доля.
- MIO 42% → 8.86% (не bottleneck), но и вклад L/D reads очень мал.
- Headroom: 8704 − 8192 (smQ prefetch если реализуем) = 512 B ← ровно на L/D prefetch.

**Диагноз (b)**: причина смерти "MIO-bound" ИСЧЕЗЛА, но **класс #3 = 4 ops/lane/qt = ~0.8% всех LDS**. Механическая масса **пренебрежимо мала**. **МЁРТВА глубже** — независимо от причины, потенциал сотые доли %.

## Могила (c) Глубина cp.async конвейера merged — паттерн 028

**Причина смерти в архиве** (028): "commit → wait<0> → sync — нулевая работа между инструкциями, глубина конвейера = 1 (single wave prefetch)".

**Проверка на текущем коде**: строки 134-137 (K/V warmup) + 187-189 (Q/dO per qt):
```
cpa_commit();       // строка 187 (Q + dO commit)
cpa_wait<0>();      // строка 188 (wait ALL cp.async in flight)
__syncthreads();    // BARRIER t3 (post-loads)
```

**Паттерн 028** (commit → wait<0> → sync без работы между): **ПОДТВЕРЖДЁН точь-в-точь** — 0 работы между commit и wait, глубина конвейера = 1 wave.

**Диагноз (c)**: **КАНДИДАТ ЖИВ**.
- Механизм: `cpa_commit_group()` + prefetch следующего qt перед wait текущего → глубина ≥ 2 wave → скрытие LDG latency.
- Потолок по механизму: **long_scoreboard 6.73%** ← топ ожидающих LDG. Полное скрытие → −6.73% × 25 мс = −1.7 мс absolute (upper bound).
- Риск: тот же, что могила (a) — SMEM double-copy Q/dO нужен для prefetch. Q double = 8192 B, dO double = 16384 B ← **не влезает** в headroom 8704 B.
- **Ограничение**: без double-buffer'а smdO — глубина конвейера остаётся 1. **Могила (c) РЕАЛЬНО живёт только вместе с могилой (a) через smQ**.

## Могила (d) A' — классы #4/#6 через no-trans ldmatrix.x4

**Причина смерти в архиве 038**: "класс #4 (32 LDS.32) + класс #6 (16 LDS.32) = 48 ops/lane/qt ≈ 9% всех LDS; при mio 25.10% и других ×3 больших классах — потенциал <1% wall".

**Пересчёт по новому профилю**:
- MIO **8.86%** (не 25.10%). Общий mio_throttle стал меньше топа.
- Class #4 = 32 LDS.32, class #6 = 16 LDS.32.
- Через no-trans ldmatrix.x4.b16 (038 §4 микропроба ✓): 32 → 8 ldmatrix.x4 (класс #4), 16 → 4 ldmatrix.x4 (класс #6).
- Total ops delta: 48 → 12 = **−36 ops/lane/qt** vs предыдущий 256 → 32 для класса #7 (−224 delta).
- Пропорционально wall выигрышу 040 −12.28% на класс #7 (256 ops): 
  - Класс #7: −12.28% на −224 ops = **0.055%/op** механическая масса
  - A' 36 ops × 0.055%/op = **−2% wall потенциал** (при линейной экстраполяции, upper bound)
  - Реальный потенциал: **−1..−2% wall** (нижняя граница ABBA-порога 2/3 v2)

**Диагноз (d)**: **КАНДИДАТ**. По правилу-2/3 v2 подпадает в зону 2..3% → KEEP при медиане ≥2% И worst ≥1% ИЛИ откат при <2%. Резко зависит от точной механической массы.

- Механизм: LDS.32 → ldmatrix.x4.b16 no-trans (038 §4 100% доказана раскладка A-op)
- Потолок по механизму: 48 → 12 static ops = −36 ops = **~1-2% wall** upper (линейная экстра)
- Риск: ptxas люфт +1..+3r (252→256 держит 2 blk). Bit-exact chain 11/11.

## Могила (e) "short_sb не полировать в изоляции" (006)

**Причина смерти в архиве 006**: "short_scoreboard = ожидание LDS результата, доминирующий вклад — B-загрузки после барьера; не отделяем от общей LSU throughput. Изолированный фикс не работает без MIO-разгрузки."

**Проверка post-040**:
- short_sb: 9.71% (было 8.63%, +1.08 pp)
- MIO разгружен: 8.86% (было 25.10%, MIO-труп)
- **Причина смерти "не отделяется от MIO"** — теперь **не применима**: MIO разгружен.

**Диагноз (e)**: **причина смерти ИСЧЕЗЛА**. short_sb теперь **можно** атаковать в изоляции (в теории).

- Что конкретно генерирует short_sb 9.71%? Скорее всего:
  - LDSM.x4.trans wait (dep-chain 512B fetch — прогноз §1 040 подтверждён "short_sb ↑")
  - LDS.U16 класса #5 smV (128 ops/lane/qt fp8→fp16 conversion pipeline)
  - LDS.32 классы #4/#6 A-ops
- Механизм полировки: (i) A' цель (d) сокращает LSU→short_sb; (ii) prefetch классы #1..#7 в regs заранее — увеличивает reg pressure.
- Потолок по механизму: **9.71% × 25 мс = 2.4 мс absolute upper**. Реальный: 0.5-1 мс.

**Кандидат ЖИВ, но малой массы (< 5% wall)**.
