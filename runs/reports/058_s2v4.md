## §0. Долг — TZ 058 бумага S2v4

TZ 058 §a: "Свизл-формула писателя smQ: кандидат = element-XOR класса smdO (эталон 040) ИЛИ row-XOR ((row&7)<<4) класса smK/smV — выбрать CPU-судьей".

### §0.a Свизл-формула — CPU-судья ДВУХ кандидатов

**Кандидат A: element-XOR класса smdO 040** (формула дословно из merged Step A):
```c
int dO_xor = (i_local & 7) << 4;                              // byte-space (3-bit XOR)
cpa16(smdO_b + i_local * dO_bpr + (col_byte ^ dO_xor), ...);  // dO_bpr = Hd * sizeof(__half) = 256
```

**Кандидат B: row-XOR класса smK/smV** (формула из `fa_bwd_common.cuh` swz_byte):
```c
int swz_byte(int row, int col_bytes) {
    int chunk = col_bytes >> 4;
    int within = col_bytes & 15;
    return row * FA_BWD_STRIDE + ((chunk ^ (row & 7)) << 4) + within;   // FA_BWD_STRIDE = 128
}
```

**Наблюдение эквивалентности**:
- Кандидат A: `col_byte ^ ((row & 7) << 4)` byte-space (3-bit XOR of chunk index).
- Кандидат B: `((chunk ^ (row & 7)) << 4)` byte-space (3-bit XOR of chunk index).
- **Оба применяют XOR к chunk-index (bits 4-6 col_byte)** через `(row & 7)`.
- **Разница ЛИНЕЙНА в row_stride**: smdO Hd*2=256 (fp16), smK/smV FA_BWD_STRIDE=128 (fp8).

**Для smQ dk_new** (fp8, Hd=128 bytes/row):
- Row-stride естественно **128 bytes** — совпадает с smK/smV — **Кандидат B применим напрямую**.
- **Кандидат A** для fp8 = same formula с row_stride 128 (переиспользование `swz_byte`).

**ВЫБОР**: **Кандидат B (swz_byte)** — production-tested (smK/smV в merged/dk работают 040/033), формула унифицирована через common.cuh.

### §0.a Судья ДВУХ фаз для Кандидата B (swz_byte на smQ dk)

**Формула**: `smQ[swz_byte(i_local, col_byte)]` для writer cp.async; читатель LDSM.x2.trans.b8 сам применяет XOR.

**Фаза записи (cp.async writer)**:

Table: threads × row × chunk_index × XOR_effective → bank(byte_addr / 4 mod 32).

| Thread tid | Row (tid/8) | Chunk (tid%8) | Row XOR = (row & 7) | Swz chunk = raw XOR (row&7) | Byte_offset in row = swz_chunk * 16 | Bank(byte_offset/4 mod 32) |
|:-:|:-:|:-:|:-:|:-:|:-:|:-:|
| 0 | 0 | 0 | 0 | 0 | 0 | 0 |
| 1 | 0 | 1 | 0 | 1 | 16 | 4 |
| 7 | 0 | 7 | 0 | 7 | 112 | 28 |
| 8 | 1 | 0 | 1 | 1 | 16 | 4 |
| 15 | 1 | 7 | 1 | 6 | 96 | 24 |
| 16 | 2 | 0 | 2 | 2 | 32 | 8 |
| ... | ... | ... | ... | ... | ... | ... |

**Bank distribution для 32-lane warp (tid 0..31)**:
- Rows 0..3 (thread groups 8), each covers 8 chunks
- Chunks with XOR by (row&7) → permutation per row
- Overall 32 threads hit 8 unique banks (0, 4, 8, 12, 16, 20, 24, 28) — **8-way structural distribution**

**Вердикт writer bank-фазы: ЗЕЛЁНАЯ ✓** (cp.async LDGSTS.E.BYPASS обходит explicit conflict metric; structurally 8-way spread).

**Фаза чтения (LDSM.x2.trans.b8 reader)**:

Row_ptr per lane = `&smQ[swz_byte(lane_row, lane_col)]` где lane_row = layout из 049-B lane-shift.

**Row_ptr formula (049-B in-bounds)**:
- `sm_addr_lo = &smQ[(kb*32 + lane) * Hd + np*16]` (для b0-pair)
- `sm_addr_hi = &smQ[(kb*32 + (lane & 15) + 16) * Hd + np*16]` (для b1-pair)

**СО СВИЗЛОМ**: row_ptr = `&smQ[swz_byte(row, np*16)]` где row = (kb*32 + lane) для lo, (kb*32 + (lane&15) + 16) для hi.

**Address-фаза analysis**:
- Row (kb*32 + lane) для lo: rows 0..31 при kb=0, 32..63 при kb=1.
- Row & 7 = (kb*32 + lane) & 7 = lane & 7 (since 32 mod 8 = 0).
- Chunk_col = np * 16 / 16 = np ∈ [0..7].
- Swizzled chunk index = np ^ (lane & 7).
- Byte_offset within row = (np ^ (lane & 7)) * 16.
- Bank(row * 128 + byte_offset) / 4 mod 32:
  - row * 128 / 4 = row * 32 mod 32 = 0 (rows 0..31 all → bank column 0).
  - byte_offset / 4 mod 32 = (np ^ (lane & 7)) * 4 mod 32.
- **Bank per lane** = `((np ^ (lane & 7)) * 4) mod 32`.

**Для fixed np (LDSM cooperative reads один np at a time)**:
- 32 lanes with lane & 7 ∈ {0..7} × 4 groups
- (np ^ (lane & 7)) permutes chunks 0..7
- (permuted) * 4 mod 32 ∈ {0, 4, 8, 12, 16, 20, 24, 28} — 8 unique banks
- Every 4 lanes (same lane & 7) hit same bank quad → **4-way conflict per LDSM instruction**

**vs 051 без свизла**: rows all → bank 0 → **32-way storm** (×5.07 events, 132M → 8189M).

**Со свизлом (кандидат B)**: **4-way conflict** (вместо 32-way), **8× reduction в bank conflict events**.

**Вердикт reader address-фазы: ЗЕЛЁНАЯ (4-way conflict, структурный пол)** ✓
- Не **0 events** (structural pol от bank stride 32), но **8× reduction** vs storm 051.

**Данная фаза (waves per охапка)**:
- LDSM.x2.trans.b8 = 2 матрицы × 16×16 fp8 = 512 fp8 = 512 bytes охапка.
- Row-stride 128B → 512B / 128B = **4 waves per x2** (structural пол).
- Wavefronts per instruction: **4/x2** (пол подтверждён 043 §1 таблица #10 compile+run ✓).

**Итог §0.a**: Кандидат B ЗЕЛЁНЫЙ обе фазы (writer 8-way, reader 4-way + структурный пол данной 4/x2). **Storm 051 не возвращается** (8× reduction гарантирована).

### §0.d Расписание LDSM внутри np-петли + liveness

**План** (уроки 048-050 + 056 lifetime compaction):
```c
for (int kb = 0; kb < KB_DK; ++kb) {
    for (int np = 0; np < NI_DK / 2; ++np) {
        int ni_a = 2*np, ni_b = 2*np + 1;
        // LDSM.x2.trans.b8 loads B0a, B0b, B1a, B1b (b0/b1 halves for MMA-B)
        uint32_t B0a, B0b, dm1, dm2;
        uint32_t sm_addr_lo = __cvta_generic_to_shared(
            &smQ[swz_byte(kb*32 + lane, np*16)]);
        asm volatile("ldmatrix.sync.aligned.m16n16.x2.trans.shared.b8 {%0,%1,%2,%3},[%4];"
            : "=r"(B0a), "=r"(B0b), "=r"(dm1), "=r"(dm2) : "r"(sm_addr_lo));
        uint32_t B1a, B1b, dm3, dm4;
        uint32_t sm_addr_hi = __cvta_generic_to_shared(
            &smQ[swz_byte(kb*32 + (lane & 15) + 16, np*16)]);
        asm volatile("ldmatrix.sync.aligned.m16n16.x2.trans.shared.b8 {%0,%1,%2,%3},[%4];"
            : "=r"(B1a), "=r"(B1b), "=r"(dm3), "=r"(dm4) : "r"(sm_addr_hi));
        // 2 MMA calls (ni_a, ni_b) with B0a/B1a and B0b/B1b
        mma_m16n8k32_e4m3_f32(dK_acc[ni_a][0..3], A0..A3, B0a, B1a, dK_acc[ni_a][0..3]);
        mma_m16n8k32_e4m3_f32(dK_acc[ni_b][0..3], A0..A3, B0b, B1b, dK_acc[ni_b][0..3]);
    }
}
```
